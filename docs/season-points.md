Передача логики season points и иерархии из текущего diff в `main`

Цель
Этот документ объясняет логику текущего незакоммиченного diff в `main`, чтобы то же поведение можно было заново реализовать или сохранить в будущей ветке с рефакторингом, не перечитывая весь патч.

Этот diff не является одной изолированной правкой. Он меняет четыре связанные области:

Парсинг в worker теперь выводит `DNF`, когда payload раунда Disc Golf Metrix структурно неполный.
Загрузка иерархии в API теперь разворачивается и вверх к родителям, и вниз к sibling/descendant сущностям, чтобы определение owner competition оставалось корректным, когда стартовый набор содержит только часть раундов.
Начисление season points теперь выбирает строку матрицы по `players_count - unique_dnf_players`, а не по сырому числу ranked finisher'ов и не по исходному размеру соревнования, если некоторые игроки явно не финишировали.
Генератор и SQL-миграция обновляют матрицу season points для сезонов 2025 и 2026.

Почему появился этот diff
Смешивались две разные проблемы скоринга:

Некоторые payload'ы результатов Metrix явно не помечали `DNF`, хотя в `PlayerResults` отсутствовала одна или несколько корзин.
Lookup season points мог попадать в неправильную строку матрицы, потому что слой скоринга использовал неверный effective field size.

Ожидаемое поведение после этого diff:

отсутствующие результаты по лункам должны трактоваться как `DNF`
игроки с `DNF` не должны считаться валидными участниками поля для matrix lookup
при этом matrix lookup не должен схлопываться просто до количества ranked finisher'ов, если на событии игроков было больше
определение owner competition для дочерних раундов должно оставаться согласованным с list/detail-моделью иерархии, даже если изначально загружены только раунды одного пула

Новое правило
Когда payload содержит известное число tracks/holes, а коллекция `PlayerResults` у игрока короче этого количества track'ов или содержит пустой результат для любой ожидаемой лунки, игрок должен считаться `DNF`.

Как это реализовано
`readResultEntries` теперь:

считывает количество track'ов из `Competition.Tracks` или из верхнеуровневого `Tracks`
читает результаты игрока по лункам из `PlayerResults`, `playerResults` или `player_results`
считает лунку завершенной только если `Result`/`result`/`Score`/`score` присутствует и не пустой
добавляет `DNF: true` в спроецированную запись, если структура раунда неполная

Важный инвариант
Это не эвристика по месту. Код не выводит `DNF` из rank, total score или отсутствующего класса. Он делает вывод только из неполных данных по отдельным лункам, когда известно ожидаемое количество track'ов.

Небольшая деталь парсера
`Place` и `place` были добавлены в `RELEVANT_RESULT_RECORD_FIELDS`. Это сохраняет информацию о placement в наборе спроецированных записей, когда она присутствует в исходном payload.

Проблема
Старые загрузчики иерархии в `players` и `results` поднимались только вверх от seed competitions к родителям.

Это ломает owner resolution, когда вызывающий код засевает иерархию только частью дочерних раундов. Пример:

`Event event-100`
Пулы `pool-long` и `pool-short`
Раунды под каждым пулом

Если seed содержит только `round-long-1` и `round-long-2`, проход вверх находит `pool-long` и `event-100`, но не узнает, что у `event-100` несколько пулов. Без этого знания о sibling'ах resolver owner'а может ошибочно поднять эти раунды до `event`, вместо того чтобы оставить их принадлежащими `pool-long`.

Новое поведение загрузчика
`loadCompetitionHierarchyContext` теперь делает два прохода:

Разворачивание вверх: загружает seed competitions, затем продолжает загружать их родителей, пока цепочка не закончится.
Разворачивание вниз: для каждой загруженной competition загружает ее children по `parent_id`, затем рекурсивно разворачивает и этих children тоже.

Это создает локальное замыкание иерархии вокруг seed-набора, а не только цепочку родителей.

Почему важен проход вниз
Owner resolution зависит от знания, есть ли у события:

один пул, тогда раунды все еще могут резолвиться к owner'у события
несколько пулов, тогда раунды должны оставаться привязанными к owner'у конкретного пула

Тесты, добавленные в `apps/api/src/app.test.ts`, фиксируют оба случая:

событие с несколькими пулами, засеянное раундами только одного пула, должно резолвиться к owner'у пула
событие с одним пулом, засеянное строками раундов, должно резолвиться к owner'у события

Замечание для рефакторинга
Новый общий загрузчик не является извлечением только ради оптимизации. Семантическое изменение здесь именно в разворачивании descendant'ов. Рефакторинг, который оставит только общий helper, но уберет второй проход, вернет баг обратно.

Выбор строки матрицы season points
Файлы:

`apps/api/src/modules/season-standings/index.ts`
`apps/api/src/app.test.ts`

Старое поведение
`resolveSeasonPointsPlayersCount` возвращал:

`max(normalizedCompetitionPlayersCount, rankedResultsCount)`

Это сохраняло большее из официального размера соревнования и количества ranked finisher'ов, но игнорировало участников с явным `DNF`.

Новое поведение
`resolveSeasonPointsPlayersCount` теперь получает полную коллекцию `competitionResults` и возвращает:

`max(normalizedCompetitionPlayersCount - unique_dnf_players, rankedResultsCount)`

где `unique_dnf_players` — это количество уникальных значений `player_id` с `dnf === true`.

Почему это изменение нужно
Если в соревновании было 42 игрока, но один игрок имеет `DNF`, эффективная строка матрицы должна быть 41, а не:

42, потому что игрок с `DNF` не должен раздувать effective field для season points
и не 3, если в уменьшенной выборке ranked остались только три игрока

Тесты обновлены так, чтобы зафиксировать случай `42 -> 41`:

очки за 1 место становятся `78.5` вместо `80`
очки за 2 место становятся `70` вместо `72`
очки за 3 место становятся `64` вместо `65`
и итоговые season totals меняются соответственно

Важный инвариант
Этот diff намеренно вычитает только игроков с явным `DNF`. Он не вычитает каждого отсутствующего финишера и не пересчитывает размер поля только по `rankedResults.length`.

Это различие важно, потому что правила проекта требуют, чтобы matrix lookup оставался привязан к официальному размеру scoring competition, кроме теперь поддержанного случая, когда игроки явно известны как `DNF`.

Что было добавлено
Небольшой generator script преобразует треугольную CSV-матрицу в SQL, который:

убеждается, что сезоны 2025 и 2026 существуют
удаляет существующие строки `season_standings` для этих сезонов
удаляет существующие строки `season_points_table` для этих сезонов
вставляет одну и ту же сгенерированную матрицу для обоих сезонов

В `package.json` добавлено:

`generate:season-points-migration`

чтобы генератор можно было снова запускать из `npm`-скриптов.

Правила парсинга CSV
Генератор ожидает:

строку заголовка с колонками `players_count` после первой ячейки
в первом столбце каждой строки данных должно быть placement
треугольные данные: пустые ячейки допустимы только когда `placement > players_count`

Он немедленно падает с ошибкой, если:

ширина строки не совпадает с заголовком
отсутствует обязательное значение points
значение points присутствует вне допустимой треугольной области
парсинг integer или point завершился ошибкой

Важная оговорка по тесту
`scripts/generate-season-points-migration.test.mjs` сейчас читает fixture по абсолютному пути в `~/Downloads`.

В текущем локальном окружении этот тест проходит, но в таком виде он непереносим. Если эту логику планируется сохранять надолго, fixture следует перенести в репозиторий.

Правило синхронизации веток, использованное здесь
При переносе этого изменения в deploy-специфичные ветки:

`main` должен сохранить полный продуктовый diff, включая `package.json`
`amvera-api` и `amvera-web` должны получить код, тесты, миграцию и handoff-документ
шум из `config/service` должен остаться вне этих веток
если "without config" трактуется строго, исключай `package.json` из branch-sync шага, даже несмотря на то, что он полезен в `main`

Минимальная ожидаемая проверка после переноса
Как минимум должны сохраниться следующие результаты:

Неполный `PlayerResults` приводит к выводу `DNF`.
События с несколькими пулами, засеянные подмножеством раундов, по-прежнему резолвятся к ownership пула.
События с одним пулом, засеянные раундами, по-прежнему резолвятся к ownership события.
Lookup матрицы season points использует `competition_players_count - unique_dnf_players`.
Генератор миграции по-прежнему выдает тот же SQL матрицы для двух сезонов.
