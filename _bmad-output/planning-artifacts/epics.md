# metrixParser - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for metrixParser, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

### NonFunctional Requirements

NFR1: Страницы списков должны открываться достаточно быстро, чтобы пользователь не воспринимал интерфейс как тормозящий в обычной работе.
NFR2: Запуск обновления должен срабатывать без ощутимой задержки после действия пользователя.
NFR3: После завершения обработки пользователь должен быстро получать итоговую статистику обновления.
NFR4: Если в интерфейсе появляются поиск и фильтрация, они должны работать с предсказуемой и комфортной скоростью на типовых объёмах данных MVP.
NFR5: Повторный запуск любого сценария обновления должен быть безопасным и не требовать ручной очистки данных перед повторной обработкой.
NFR6: После сбоя или частичной проблемы пользователь должен понимать, что именно произошло, по отображаемому результату операции и статистике обновления.
NFR7: Система не должна создавать ситуацию, в которой ошибка в отдельных записях делает состояние всей операции полностью непрозрачным для пользователя.
NFR8: Интерфейс должен обеспечивать базовую аккуратную веб-доступность без формальной сертификации.
NFR9: Интерфейс должен включать читаемый текст, понятные формы, достаточную контрастность и возможность базовой работы с клавиатуры в основных пользовательских сценариях.
NFR10: Интеграция с DiscGolfMetrix должна корректно обрабатывать ошибки API и отражать их в понятной форме для пользователя или владельца проекта.
NFR11: Система должна поддерживать безопасный повторный запуск загрузки после ошибок интеграции без необходимости ручного восстановления данных.
NFR12: Поведение интеграции должно оставаться предсказуемым при неполных или проблемных ответах внешнего источника.

### Additional Requirements

- Архитектура должна использовать `Vite + React + TypeScript` как frontend foundation.
- Архитектура должна использовать `Supabase Postgres` как единственную persistent database на MVP.
- Архитектура должна включать отдельный backend API как единственную backend boundary для UI.
- Архитектура должна включать отдельный TypeScript worker для import-операций.
- Frontend, API и worker должны деплоиться как отдельные runtime-контура.
- Frontend не должен обращаться к БД напрямую и не должен содержать import-логику.
- Frontend должен работать только через backend API.
- Worker должен быть изолирован от frontend runtime и выполнять import-процессы как отдельный процесс.
- API должен принимать команды запуска обновлений, читать данные для UI и возвращать статистику и диагностическую информацию.
- DB schema naming должно использовать `snake_case`.
- API and frontend boundary должны использовать `camelCase`.
- API success response должен использовать формат `{ data, meta }`.
- API error response должен использовать формат `{ error: { code, message } }`.
- Frontend должен быть организован по принципу `feature-first`.
- Backend API и worker должны быть организованы по принципу `layer-first`.
- Import logic, API handlers, data access, mapping and validation не должны смешиваться в одном слое.
- Ошибки импорта должны агрегироваться, а не валить весь запуск целиком.
- Проблемные записи должны маркироваться как пропущенные и отражаться в итоговой статистике.
- Loading state должен быть локален конкретной операции или экрану.
- Пользователь всегда должен видеть финальный статус выполнения и итоговую статистику.
- Репозиторий должен быть организован как монорепо с `apps/web`, `apps/api`, `apps/worker`, `packages/shared-types`, `packages/shared-utils` и `supabase/`.
- Не зафиксирован точный orchestration mechanism между `apps/api` и `apps/worker`; этот вопрос должен быть уточнён до начала реализации update flows.

## Epic 1: Базовый контур платформы и ручной запуск обновлений

Пользователь получает работающий каркас системы: SPA, backend API, worker, схему данных и возможность вручную запускать обновления с понятным итоговым статусом выполнения.

### Story 1.1: Инициализация монорепозитория и базовых приложений

As a владелец проекта,
I want иметь инициализированный монорепозиторий с приложениями `web`, `api`, `worker` и общей workspace-конфигурацией,
So that дальнейшая разработка будет вестись в согласованной архитектурной структуре.

**Acceptance Criteria:**

**Given** пустой репозиторий проекта
**When** разработчик инициализирует базовую структуру решения
**Then** в репозитории существуют `apps/web`, `apps/api`, `apps/worker`, `packages/shared-types`, `packages/shared-utils`, `supabase/`
**And** workspace-конфигурация позволяет управлять приложениями как единым монорепозиторием
**And** frontend создаётся на базе `Vite + React + TypeScript`
**And** базовые конфигурационные файлы проекта добавлены в репозиторий

### Story 1.2: Базовая интеграция Supabase и инфраструктура миграций

As a владелец проекта,
I want иметь подключённый Supabase и подготовленную инфраструктуру миграций,
So that последующие stories могли создавать доменные таблицы и безопасно развивать схему данных.

**Acceptance Criteria:**

**Given** инициализированный проект
**When** разработчик настраивает слой хранения данных
**Then** проект подключён к `Supabase Postgres`
**And** структура `supabase/` и каталог `supabase/migrations` готовы к использованию
**And** соглашения по `snake_case` для будущих таблиц и колонок зафиксированы
**And** приложение и backend могут использовать конфигурацию подключения к Supabase без прямого обращения frontend к БД

### Story 1.3: Backend API-каркас и health/read contract

As a владелец проекта,
I want иметь работающий backend API как единственную серверную границу для UI,
So that frontend не зависит напрямую от БД и import runtime.

**Acceptance Criteria:**

**Given** инициализированный backend API
**When** frontend или разработчик обращается к API
**Then** API запускается как отдельное приложение
**And** существует базовый health endpoint для проверки доступности сервиса
**And** API использует единый формат успешного ответа `{ data, meta }`
**And** API использует единый формат ошибки `{ error: { code, message } }`
**And** API готов расширяться модулями `updates`, `competitions`, `courses`, `players`, `results`

As a сотрудник РДГА,
I want видеть административный интерфейс с отдельными действиями для запуска обновлений,

**Acceptance Criteria:**

**Given** пользователь открывает SPA-приложение
**When** он переходит в административный раздел
**Then** интерфейс отображает отдельные элементы управления для обновления соревнований, парков, игроков и результатов

**And** запуск каждого сценария инициируется явным действием пользователя
**And** UI не обращается к БД напрямую, а работает через backend API

### Story 1.5: Общий lifecycle обновления и итоговая статистика

As a сотрудник РДГА,
I want после запуска обновления видеть его статус и итоговую статистику,
So that я понимаю, что произошло и можно ли доверять результату.

**Acceptance Criteria:**

**Given** пользователь запускает любой сценарий обновления
**When** операция начинается
**Then** UI отображает локальный loading state для конкретной операции
**And** после завершения пользователь видит финальный статус выполнения
**And** интерфейс отображает количество найденных, добавленных, обновлённых и пропущенных записей
**And** формат результата одинаков для всех сценариев обновления

As a владелец проекта,
I want чтобы повторный запуск обновлений был безопасным и устойчивым к частичным ошибкам,

**When** пользователь повторно запускает обновление
**Then** система не создаёт дубли по согласованным идентификаторам
**And** система отличает новые записи от обновляемых
**And** корректные записи продолжают обрабатываться даже при наличии отдельных ошибочных или неполных записей
**And** проблемные записи помечаются как пропущенные
**And** пользователь видит результат частичной обработки в статистике обновления

## Epic 2: Импорт и сохранение данных о соревнованиях и парках

Пользователь может получать и сохранять данные о соревнованиях и парках из DiscGolfMetrix в структурированном виде, с корректным обновлением существующих записей.

### Story 2.1: Интеграция worker с DiscGolfMetrix для получения соревнований

As a владелец проекта,
I want чтобы worker мог получать данные о соревнованиях из DiscGolfMetrix за заданный период,
So that система могла запускать сценарий обновления соревнований на основе внешнего источника данных.
