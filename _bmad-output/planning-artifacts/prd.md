# Product Requirements Document - metrixParser

**Author:** Darling
**Date:** 2026-03-18 12:42:16

## Executive Summary

RU Disc Golf Parser - это веб-приложение для ручного запуска, сбора, сохранения и последующего просмотра данных о российских дискгольф-соревнованиях из DiscGolfMetrix. Продукт создаётся как greenfield-решение, чтобы устранить текущий полностью ручной процесс, при котором сотрудникам приходится просматривать календарь соревнований, открывать каждое событие отдельно и вручную переносить данные с экранов в рабочие материалы.

Основная ценность продукта - резкое сокращение трудозатрат на сбор данных. Вместо нескольких часов или дней ручной работы пользователь должен получать структурированные данные по соревнованиям, паркам, игрокам и результатам через единый интерфейс за несколько минут. Первая версия фокусируется на ручном запуске обновлений и получении данных по четырём ключевым сущностям, без усложнения в виде автоматических расписаний или избыточной аналитики.

Целевым пользователем продукта является руководство РДГА и другие участники процесса, которым нужны актуальные данные о соревнованиях, но у которых нет быстрого способа их собрать. Продукт решает не абстрактную задачу аналитики, а конкретную операционную проблему: отсутствие удобной выгрузки и высокая стоимость ручного сбора информации.

### What Makes This Special

Продукт отличается тем, что закрывает очень конкретную и дорогостоящую по времени проблему в узком домене. Его преимущество не в необычном интерфейсе, а в том, что он превращает разрозненный, медленный и ручной процесс в короткий управляемый сценарий через админку. Пользователь задаёт параметры, запускает нужное обновление и получает понятный результат с сохранением данных в структурированном виде.

Ключевой инсайт проекта состоит в том, что для MVP важнее не продвинутая аналитика, а надёжное и быстрое получение базовых сущностей. Именно скорость доступа к данным и устранение ручного копирования создают немедленную ценность для РДГА и делают продукт лучше любых обходных способов работы с DiscGolfMetrix.

## Success Criteria

### User Success

Пользователь должен иметь возможность за несколько минут получить актуальные данные по `competitions`, `courses`, `players` и `competition_results` через единый интерфейс без ручного копирования данных из DiscGolfMetrix. MVP считается полезным, если пользователь может самостоятельно запустить нужные обновления и получить результат в структурированном виде за один короткий рабочий сеанс.

### Business Success

С точки зрения РДГА продукт считается успешным, если он заменяет многодневный ручной сбор данных коротким управляемым процессом. Главный бизнес-эффект MVP - высвобождение времени руководства и сокращение операционных затрат на сбор и подготовку данных о российских дискгольф-соревнованиях.

### Technical Success

Система должна обеспечивать стабильный ручной запуск трёх независимых сценариев обновления: соревнований, парков, игроков и результатов. Данные должны сохраняться без дублей, без необоснованных пропусков и без разрушения общего процесса при наличии битых записей. После каждого запуска пользователь должен видеть понятную статистику обновления, достаточную для контроля результата.

### Measurable Outcomes

- Пользователь получает актуальные данные по четырём ключевым сущностям за несколько минут, а не за часы или дни ручной работы.
- Один короткий сеанс работы через интерфейс заменяет полный цикл ручного поиска, открытия соревнований и копирования данных.
- Все три сценария ручного обновления выполняются независимо друг от друга и завершаются с понятным результатом.
- Ошибочные или битые записи не приводят к остановке всего обновления.

## Product Scope

### MVP - Minimum Viable Product

- Ручной запуск обновления `competitions` по заданному периоду.
- Ручной запуск обновления `courses` на основе данных о соревнованиях.
- Ручной запуск обновления `players/results` по соревнованиям за заданный период.
- Стабильная запись данных в таблицы `competitions`, `courses`, `players`, `competition_results`.
- Защита от дублей и необоснованных пропусков.
- Отображение понятной статистики обновления после каждого запуска.

### Growth Features (Post-MVP)

- Автоматизация расписания сбора данных.
- Защита административной части паролем.

### Vision (Future)

- Расчёт рейтинга игроков по фиксированной формуле на основе собранных данных.

## User Journeys

### Journey 1 - Оператор РДГА запускает обновление данных

Сотрудник РДГА или его помощник открывает административный интерфейс, потому что ему нужно быстро собрать актуальные данные о соревнованиях без ручного обхода DiscGolfMetrix. Он задаёт период или запускает нужный тип обновления, выбирая один из трёх независимых сценариев: соревнования, парки, игроки и результаты. Ключевой момент ценности наступает тогда, когда через несколько минут система завершает обработку и показывает понятную статистику: что найдено, что добавлено, что обновлено и что пропущено. Вместо многочасового ручного копирования пользователь получает управляемый и повторяемый рабочий сценарий.

### Journey 2 - Оператор РДГА повторно запускает обновление и контролирует качество данных

Тот же пользователь возвращается в систему, когда нужно обновить уже частично собранные данные или проверить корректность очередного запуска. Он повторно запускает обновление и ожидает, что система не создаст дубли, не пропустит записи без причины и не сломается из-за отдельных битых данных. Успешный сценарий для него - увидеть, что повторный запуск отрабатывает стабильно, статистика остаётся понятной, а данные в базе ведут себя предсказуемо. Этот путь раскрывает требования к идемпотентности, обработке ошибок и прозрачной диагностике.

### Journey 3 - Руководитель или сотрудник РДГА использует собранные данные для работы со статистикой

После завершения обновления руководитель РДГА или другой сотрудник открывает интерфейс просмотра данных, чтобы использовать уже собранную информацию в дальнейшей работе. Ему не нужно знать детали интеграции с DiscGolfMetrix; для него важно, что данные уже собраны, структурированы и доступны по ключевым сущностям. Он просматривает соревнования, парки, игроков и результаты, чтобы на их основе считать статистику и принимать дальнейшие рабочие решения. Ценность продукта для этого пользователя в том, что данные становятся доступными без промежуточного ручного сбора.

### Journey 4 - Игрок находит свои результаты через интерфейс

Игрок в дискгольф заходит в интерфейс как потребитель данных, а не как оператор системы. Его задача проста: найти свои результаты и быстро понять, как он выступил. Для него продукт полезен, если результаты доступны в понятном виде и не требуют самостоятельного поиска по внешним источникам или ручной сверки. Этот сценарий раскрывает требования к публичному просмотру данных и простоте навигации по результатам.

### Journey 5 - Владелец проекта разбирает сбой или подозрительные пропуски

Если после обновления появляются пропуски, странные результаты или проблемы обработки, владелец проекта выступает как support и troubleshooting-role. Он не пользуется системой как массовый конечный пользователь, но должен быстро понять, что именно произошло во время запуска. Для него критично, чтобы система не завершалась молча, а оставляла понятную статистику и признаки того, где возникла проблема. Этот путь раскрывает требования к наблюдаемости, понятной диагностике и устойчивости процесса обновления.

### Journey Requirements Summary

- Система должна поддерживать отдельный административный путь для ручного запуска трёх независимых сценариев обновления.
- Система должна быть устойчивой к повторным запускам и не создавать дубли при повторной обработке данных.
- Система должна показывать понятную статистику результата каждого запуска для оператора и для отладки.
- Система должна предоставлять интерфейс просмотра собранных данных для внутренних пользователей РДГА.
- Система должна предоставлять внешний сценарий просмотра результатов для игроков.
- Система должна оставлять достаточно диагностической информации, чтобы владелец проекта мог разбирать ошибки и пропуски.

## Domain-Specific Requirements

### Data Truth

Система рассматривает `DiscGolfMetrix` как единственный источник истины. Все данные о соревнованиях, парках, игроках и результатах должны интерпретироваться как производные от актуального состояния данных в DiscGolfMetrix. Ручные корректировки данных внутри системы в рамках MVP не предусматриваются.

### Technical Constraints

Система не должна сохранять частичные результаты соревнований. Если отдельная запись результата не может быть корректно разобрана или пришла в неполном виде, она не должна попадать в итоговые таблицы как валидный результат. Обработка `DNF` должна сохраняться как отдельный логический признак результата и учитываться в доменной логике корректно.

### Identity and Matching

Для повторных обновлений и предотвращения дублей система должна использовать устойчивые идентификаторы из DiscGolfMetrix:
- `competition_id` и `metrix_id` для соревнований
- `player_id` для игроков

Эти идентификаторы рассматриваются как надёжные ключи сопоставления при повторной загрузке и обновлении существующих записей.

### Operational Risks

Основные предметные риски в домене связаны не с compliance, а с качеством входных данных и предсказуемостью интеграции:
- часть результатов может прийти, а часть отсутствовать;
- код может не разобрать отдельные результаты;
- отдельные записи могут быть неполными или битым образом представлены в ответе API.

### Risk Mitigations

Система должна сохранять все корректно разобранные данные и не останавливать весь процесс из-за отдельных проблемных записей. Неразобранные, неполные или ошибочные записи должны помечаться как пропущенные и отражаться в статистике обновления, чтобы пользователь и владелец проекта могли контролировать качество загрузки и разбирать проблемные случаи.

## Web App Specific Requirements

### Project-Type Overview

Продукт реализуется как SPA-веб-приложение с разделением на административный интерфейс для запуска обновлений и пользовательские страницы просмотра данных. Основной пользовательский сценарий предполагает работу через современные desktop-браузеры. Мобильная адаптация может быть полезной, но не является определяющим требованием MVP.

### Technical Architecture Considerations

Система не требует real-time-обновления интерфейса. Для MVP достаточно ручного запуска операций и обычного обновления страницы или повторного запроса данных после завершения обработки. Это позволяет упростить архитектуру интерфейса и избежать лишней сложности в клиент-серверном взаимодействии.

SEO не является требованием MVP, так как ключевая ценность продукта связана не с поисковым трафиком, а с доступом к данным и внутренними рабочими сценариями. Поэтому архитектура интерфейса может оптимизироваться под продуктовую простоту и скорость разработки, а не под индексируемость страниц.

### Browser and Accessibility Requirements

MVP должен корректно работать в современных desktop-браузерах. Формальная сертификация по accessibility не требуется, но интерфейс должен соответствовать базовым принципам аккуратной веб-доступности: читаемые элементы, понятные формы, логичная навигация, различимые состояния интерфейса.

### Implementation Considerations

Выбор SPA-подхода должен поддерживать быстрое создание админского интерфейса, понятную навигацию между страницами данных и простую интеграцию с backend-операциями ручного запуска обновлений. Отсутствие требований по SEO и real-time позволяет сконцентрироваться на скорости реализации, предсказуемости интерфейса и удобстве работы основных пользователей.

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** problem-solving MVP с немедленной прикладной ценностью. Первая версия должна доказать, что система может быстро собирать данные из DiscGolfMetrix и сразу делать их доступными через удобный интерфейс просмотра.

**Resource Requirements:** минимальная solo-команда в составе владельца проекта с использованием AI-assisted development workflow (`VS Code + Codex CLI + GPT-5.4`). Scope MVP должен учитывать крайне ограниченный человеческий ресурс и опираться на простую, быструю в реализации архитектуру.

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- оператор РДГА вручную запускает обновление данных;
- оператор получает понятную статистику результата обновления;
- внутренний пользователь РДГА просматривает собранные данные через интерфейс;
- игрок просматривает свои результаты через интерфейс.

**Must-Have Capabilities:**
- ручной сбор и сохранение данных по 4 сущностям: `competitions`, `courses`, `players`, `competition_results`;
- три независимых сценария обновления через административный интерфейс;
- отображение статистики после каждого обновления;
- отображение собранных данных на соответствующих страницах интерфейса.

### Post-MVP Features

**Phase 2 (Post-MVP):**
- автоматизация расписания сбора данных;
- защита админки паролем;
- улучшения удобства просмотра и навигации по данным.

**Phase 3 (Expansion):**
- расчёт рейтинга игроков по фиксированной формуле;
- расширенная статистика и производные аналитические представления;
- дальнейшее развитие продукта как базы для аналитики дискгольф-результатов.

### Risk Mitigation Strategy

**Technical Risks:**
- интеграция с DiscGolfMetrix;
- корректность и полнота входных данных;
- устойчивость разбора данных при неполных или проблемных ответах.

**Mitigation Approach:**
- строить MVP вокруг простых независимых сценариев обновления;
- использовать устойчивые идентификаторы и идемпотентное сохранение;
- сохранять корректные записи даже при частичных проблемах;
- отражать пропуски и ошибки в статистике обновления.

**Market Risks:**
- риск низкий, так как проблема уже подтверждена существующим ручным трудом и конкретной болью РДГА.

**Validation Approach:**
- успех MVP определяется тем, что ручной многодневный процесс заменяется коротким сеансом работы через интерфейс.

**Resource Risks:**
- проект реализуется в условиях минимального человеческого ресурса.

**Contingency Approach:**
- держать MVP узким;
- откладывать автоматизацию, защиту админки и производную аналитику на post-MVP этапы;
- избегать архитектурных решений, не влияющих напрямую на ценность первой версии.

## Functional Requirements

### Data Update Management

- FR1: Оператор РДГА может запускать обновление соревнований за указанный период.
- FR2: Оператор РДГА может запускать обновление парков независимо от обновления соревнований.
- FR3: Оператор РДГА может запускать обновление игроков и результатов за указанный период независимо от других обновлений.
- FR4: Система может выполнять каждый из трёх сценариев обновления как отдельную независимую операцию.
- FR5: Система может использовать данные соревнований как источник для определения парков, подлежащих обновлению.
- FR6: Система может использовать сохранённые соревнования за выбранный период как источник для обновления игроков и результатов.
- FR7: Оператор РДГА может повторно запускать любой сценарий обновления без необходимости ручной очистки данных.

### Competition Data Management

- FR8: Система может получать данные о соревнованиях из DiscGolfMetrix.
- FR9: Система может отбирать для обработки соревнования, относящиеся к России.
- FR10: Система может сохранять новые записи соревнований.
- FR11: Система может обновлять ранее сохранённые записи соревнований.
- FR12: Система может хранить для соревнования `competition_id`, `competition_name`, `competition_date`, `course_name`, `record_type`, `players_count`, `metrix_id`.

### Course Data Management

- FR13: Система может получать данные о парках на основе идентификаторов, обнаруженных в данных соревнований.
- FR14: Система может сохранять новые записи парков.
- FR15: Система может обновлять ранее сохранённые записи парков.
- FR16: Система может рассчитывать и сохранять суммарный `course_par` парка.
- FR17: Система может хранить для парка `id`, `name`, `fullname`, `type`, `country_code`, `area`, `rating_value1`, `rating_result1`, `rating_value2`, `rating_result2`, `course_par`.

### Player and Result Data Management

## Non-Functional Requirements

### Performance

Страницы списков должны открываться достаточно быстро, чтобы пользователь не воспринимал интерфейс как тормозящий в обычной работе. Запуск обновления должен срабатывать без ощутимой задержки после действия пользователя. После завершения обработки пользователь должен быстро получать итоговую статистику обновления. Если в интерфейсе появляются поиск и фильтрация, они должны работать с предсказуемой и комфортной скоростью на типовых объёмах данных MVP.

### Reliability

Повторный запуск любого сценария обновления должен быть безопасным и не требовать ручной очистки данных перед повторной обработкой. После сбоя или частичной проблемы пользователь должен понимать, что именно произошло, по отображаемому результату операции и статистике обновления. Система не должна создавать ситуацию, в которой ошибка в отдельных записях делает состояние всей операции полностью непрозрачным для пользователя.

### Accessibility

Интерфейс должен обеспечивать базовую аккуратную веб-доступность без формальной сертификации. Это включает читаемый текст, понятные формы, достаточную контрастность и возможность базовой работы с клавиатуры в основных пользовательских сценариях.

### Integration

Интеграция с DiscGolfMetrix должна корректно обрабатывать ошибки API и отражать их в понятной форме для пользователя или владельца проекта. Система должна поддерживать безопасный повторный запуск загрузки после ошибок интеграции без необходимости ручного восстановления данных. Поведение интеграции должно оставаться предсказуемым при неполных или проблемных ответах внешнего источника.
